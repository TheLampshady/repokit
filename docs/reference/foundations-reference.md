# Foundations — Reference

Background research for repokit's "foundations" concept: what to call the shared, testable, decoupled, DRY building blocks at the base of a codebase, how teams document them, how they detect when those abstractions go bad, and who owns them.

This doc is reference material for the design of:

- a `FOUNDATIONS.md` registry generated by **dockit**
- per-foundation subject-matter-expert subagents generated by **agentkit**
- foundation-health signals emitted by the **auditor** agent
- foundation-related tickets written by **modernizer** (in the sibling [tikkit](https://github.com/TheLampshady/tikkit) plugin)

It does not specify implementation. It captures the terminology and prior art so design decisions have a paper trail.

---

## 1. Terminology

The industry has many words for "shared, foundational code." None is universal. The table below summarises which terms carry which meaning so we can pick deliberately.

| Term | Origin / community | What it names | Flavour | Verdict |
|------|--------------------|---------------|---------|---------|
| **Foundation** | Informal | Role in the codebase | Neutral | **User-facing label.** Already adopted by repokit. |
| **Core / `core/`** | Universal directory convention | An implementation location | Neutral | A hint, not the concept. |
| **Shared Kernel** | DDD (Eric Evans) | Pattern + entity that two contexts deliberately share | Backend-leaning | **Borrow the discipline.** DDD's rule: keep it small; growth is a smell that contexts should be merged or split. |
| **Component** (Backstage) | Platform engineering | A registry entry (one row in a software catalog) | Neutral | **Adopt as registry vocabulary.** Backstage's `kind: Component` with `spec.type`, `spec.lifecycle`, `spec.owner` is the cleanest schema in the wild. |
| **Software Catalog** | Backstage / Spotify | The registry itself | Neutral | **Adopt as the index concept.** `FOUNDATIONS.md` is a catalog. |
| **Paved Road / Golden Path** | Netflix / Spotify | Templates for *building new things* | Neutral | Different concept. About starting points, not existing shared code. |
| **Platform code** | Team Topologies | Code owned by a Platform team | Neutral | Implies a team boundary; useful for ownership only. |
| **Ports & Adapters / Hexagonal** | Cockburn | Boundary technique | Backend-leaning | A pattern for *building* foundations, not a registry. |
| **Anti-corruption layer** | DDD | Specific subtype of boundary | Backend | Subtype only. |
| **Design system** | Frontend | Visual + component foundation | Frontend | Frontend subset; one possible foundation type. |
| **Building block / Primitive** | Generic | Entity | Neutral | Vague but readable in prose. |
| **Module / Library** | Generic | Entity | Neutral | Too generic — every dependency is a "library." |

**Recommendation.** Use **foundation** as the noun in user-facing text. Internally model as a **Catalog** of **Components**, taking the field set from Backstage and the size discipline from DDD's Shared Kernel.

## 2. Documentation patterns

How teams in the wild write down what their foundations are.

1. **ADRs (Architecture Decision Records).** Short markdown per decision: status / context / decision / consequences. Lives in `docs/adr/`. Captures *why* a foundation exists and *why* it has its current shape — not the API surface itself. ([adr.github.io](https://adr.github.io/), [Fowler](https://martinfowler.com/bliki/ArchitectureDecisionRecord.html), [MADR template](https://adr.github.io/madr/), [joelparkerhenderson collection](https://github.com/joelparkerhenderson/architecture-decision-record))
2. **Backstage `catalog-info.yaml`.** Per-component YAML next to the code. Fields: `kind`, `spec.type`, `spec.lifecycle`, `spec.owner`, `spec.providesApis`, `spec.dependsOn`. The de-facto standard for "list of things in our system." ([Backstage Software Catalog](https://backstage.io/docs/features/software-catalog/), [System Model](https://backstage.io/docs/features/software-catalog/system-model/))
3. **Module README.** Co-located with the foundation: purpose, public API, examples, consumers, gotchas. Cheapest viable doc; doesn't depend on tooling.
4. **Storybook stories-as-docs.** Frontend foundations: API table auto-generated from props; stories double as regression and visual tests. ([storybook.js.org](https://storybook.js.org/))
5. **CODEOWNERS.** Path → team mapping. Minimum viable ownership signal. Pairs with a per-foundation `OWNERS` block in the registry.

**Synthesis — fields to collect per foundation:**

`name · type · purpose · path · public API surface · invariants · owner team · consumers (fan-in) · depends-on (fan-out) · test coverage · status (active | deprecated | sunset) · last reviewed`

This is the row schema for `FOUNDATIONS.md`.

## 3. Health signals

Measurable indicators that a foundation has drifted away from "small, testable, decoupled, DRY."

### Coupling metrics

- **Afferent coupling (Ca).** Number of consumers depending on this foundation. Foundations *should* have **high Ca** — that's the point.
- **Efferent coupling (Ce).** Number of things this foundation depends on. Foundations should have **low Ce**.
- **Instability `I = Ce / (Ca + Ce)`.** Foundations want **low I** (stable). Rising I over time is a regression signal.

References: [Software package metrics — Wikipedia](https://en.wikipedia.org/wiki/Software_package_metrics), [Afferent / Efferent Coupling — coupling.dev](https://coupling.dev/posts/related-topics/afferent-and-efferent-coupling/), [Efferent coupling — Wikipedia](https://en.wikipedia.org/wiki/Efferent_coupling).

### Size and complexity

- **God class:** ~500 LOC and/or 20+ public methods is a common threshold.
- **Cyclomatic complexity** per public method.
- **Public-API breadth:** number of exported symbols — a foundation that exports everything has no encapsulation.

### Code smells (Fowler / industry catalogues)

- **Shotgun surgery** — one logical change requires touching N call sites. Sign of a leaky abstraction. ([luzkan smells catalog](https://luzkan.github.io/smells/shotgun-surgery/))
- **Feature envy** — methods on the foundation manipulate consumer-side data more than their own.
- **Divergent change** — one foundation changes for many unrelated reasons (low cohesion).
- **Primitive obsession** — public API uses primitives where a value object would serve.

### The "wrong abstraction" signal (Sandi Metz)

Param count and conditional count growing over time on a single function indicates the abstraction *guessed wrong* about its callers' needs. Metz's prescription: *prefer duplication over the wrong abstraction*; inline it back into callers and re-extract once the real shape is visible. ([sandimetz.com — The Wrong Abstraction](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction))

### Test and review hygiene

- **Untested public surface** — public method with zero referencing test files.
- **Stale review** — last review > N days, `git log` shows touches but no review entry.

### Deprecation drift

- **Ca trending toward zero** — consumers are migrating away. Candidate for sunset. ([Heroku — Sunsetting and Deprecation](https://blog.heroku.com/sunsetting-and-deprecation), [Mastercard Client Library Deprecation Policy](https://github.com/Mastercard/.github/blob/main/CLIENT_LIBRARY_DEPRECATION_POLICY.md))

### Tools that compute these

| Language | Tool |
|----------|------|
| Cross-language | SonarQube, CodeClimate |
| JS / TS | dependency-cruiser, madge, ts-prune |
| Python | radon, vulture |
| Go | gocyclo, deadcode |

## 4. Ownership

Foundations without a named owner rot. The pattern that scales:

- **Team Topologies model.** A *Platform team* owns the foundation; *Stream-aligned teams* consume it self-service. ([teamtopologies.com](https://teamtopologies.com/key-concepts), [Fowler](https://martinfowler.com/bliki/TeamTopologies.html), [Atlassian summary](https://www.atlassian.com/devops/frameworks/team-topologies))
- **Backstage `spec.owner`** operationalises this: every component has exactly one owning team.
- **CODEOWNERS** is the minimum viable shadow of the same idea — enforced by GitHub at PR time.

**Antipattern:** "shared ownership" with no named team. Translates to no ownership in practice.

## 5. Concept-to-implementation mapping for repokit

| Concept | Repokit artefact |
|---------|------------------|
| Catalog (Backstage) | `FOUNDATIONS.md` registry generated by dockit |
| Component entry | One row in the registry |
| ADR | Existing `docs/` ADR convention (out of scope for foundations work) |
| Per-component owner | `owner` field in the row + CODEOWNERS |
| Health signal | Auditor agent dimension |
| Action on signal | Modernizer ticket in tikkit |
| Per-foundation expertise | Subagent generated by agentkit, one per row |

The split keeps each tool focused: dockit *describes*, auditor *detects*, modernizer *plans the fix*, agentkit *equips agents to work on it*.

## 6. Open questions

- **Granularity.** Is one foundation = one directory? One module? One class? Likely a mix; the registry should allow path *or* symbol scoping.
- **Thresholds.** LOC, method-count, Ca/Ce thresholds are language- and project-dependent. Where do defaults live — modernizer's language reference files, or a per-project `foundations.config`?
- **Detection without metrics tooling.** Many projects don't have SonarQube. How much can the auditor do with `git`, `grep`, and language-native tools alone?
- **Lifecycle states.** Backstage uses `experimental | production | deprecated`. Do we need a fourth state for "under refactor"?
- **Cross-plugin contract.** Auditor (repokit) emits signals; modernizer (tikkit) consumes them. What's the on-disk format — a JSON file in `specs/`? Backlog entries with a structured suffix?
