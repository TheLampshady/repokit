# agentkit

Generate, sync, and maintain project-level AI subagents that own dockit's foundations and understand your codebase's custom code patterns.

## What It Does

Reads `docs/FOUNDATIONS.md` (generated by dockit) as the source of truth, analyzes custom/extended code, and creates agents that:

- **Teach AI assistants** the rules, invariants, and conventions for each foundation
- **Update FOUNDATIONS.md** when their owned foundations change (catalog row, per-foundation entry, sub-doc)
- **Run cross-doc consistency checks** in PRINCIPLES.md and ARCHITECTURE.md when invariants or status flip

If FOUNDATIONS.md doesn't exist yet, agentkit stops and recommends `/dockit init` first.

## Modes

```
/agentkit              # Auto-detect (init / sync / status)
/agentkit init         # Generate from scratch — discover, plan, write
/agentkit sync         # Reconcile existing agents with current FOUNDATIONS.md and code
/agentkit status       # Read-only inventory and drift report
```

Optional platform filter: `/agentkit init claude`, `/agentkit sync gemini`, etc.

## How init Works

1. **Reads** `docs/FOUNDATIONS.md` — catalog rows, invariants, change checklists, sub-docs
2. **Discovers** dependencies, frameworks, custom code that extends foundations
3. **Groups** foundations into 1–5 agents using the AGENT-SIZING heuristic (combined to avoid trigger conflicts)
4. **Plans** — shows the foundation→agent mapping and asks for confirmation
5. **Generates** foundation-owner agents (with maintenance sections + edit permissions) for the chosen platforms
6. **Enriches** your instruction file (CLAUDE.md / GEMINI.md / copilot-instructions) with agent routing

## How sync Works

Reconciles existing agents against current state. Detects:

- **Orphaned agents** — claim a foundation that no longer exists
- **Missing agents** — a foundation has no owner
- **Path drift** — foundation moved without updating the agent
- **Invariant drift** — agent body and FOUNDATIONS.md disagree
- **Stale review** — `Last reviewed` date > 90 days

Prompts per finding — never auto-overwrites hand-authored agents.

## Output

Agent files are created in project-level directories:

| Platform | Location | Extension |
|----------|----------|-----------|
| Claude | `.claude/agents/` | `.md` |
| Gemini | `.gemini/agents/` | `.md` |
| Copilot | `.github/agents/` | `.agent.md` |

Plus an agent routing section appended to your existing instruction files (CLAUDE.md / GEMINI.md / copilot-instructions).

## Recommended Flow

```
/init          → Base instruction file (conventions, build commands)
/dockit init   → Project documentation, including FOUNDATIONS.md
/agentkit      → Foundation-owner + domain-expert agents, plus routing
```

**Maintenance loop:**

```
Code changes → /dockit sync → /agentkit sync
```

Or invoke a foundation-owner agent directly when only its FOUNDATIONS.md row needs updating — the agent knows the maintenance protocol.

## Agent Scaling

| Project Size | Files | Agents |
|-------------|-------|--------|
| Small | 1–20 | 1–2 |
| Medium | 21–50 | 2–4 |
| Large | 51+ | 3–5 |

Foundation count, hotspot status, and sub-doc presence influence grouping. See `references/guides/AGENT-SIZING.md` for the full mapping heuristic.

## What Makes a Good Agent

Agents own **foundations** (rows in FOUNDATIONS.md) and the custom code that extends them:
- Foundations with significant invariants and many consumers
- Custom base classes other code inherits from
- Extensions that look like framework defaults but behave differently
- Growing directories with many files following team patterns

Agents are NOT created for:
- Standard framework usage with no customization
- Foundations marked `status: sunset` or `health: pretender`
- Boilerplate that adds nothing to framework defaults
- Areas already covered by an existing hand-authored agent (warned, not replaced)
